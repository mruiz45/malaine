**Goal:** To securely save the user-defined pattern, implement the sophisticated calculation algorithms that translate the `patternState` into precise knitting/crochet instructions, and generate the textual output for the user. **User Story PD_PH6_US001: Persist Finalized Pattern Definition to Database**

* **Sequence Number:** PD_PH6_US001
* **Title:** Persist Finalized Pattern Definition (`patternState`) to Database
* **Description:** As a user, once I am satisfied with my pattern design, I want to be able to save it permanently so I can retrieve it later. As a system, we need to store the complete `patternState` in the database.
* **Functional Requirements (What & How - User Perspective):**
    1.  A "Save Pattern" button is available in the pattern definition interface.
    2.  When clicked, the current state of the pattern design (`patternState`) is saved to the user's account (assuming user authentication is in place).
    3.  The user might be prompted to give the pattern a name if not already done in `patternState.metadata.designName`.
    4.  The system should confirm successful save or report any errors.
* **Technical Implementation (What & How - Developer Perspective):**
    1.  **Backend API Endpoint:**
        * Create a new API endpoint (e.g., `POST /api/patterns` or `PUT /api/patterns/{patternId}` if updating).
        * This endpoint will accept the entire `patternState` JSON object in the request body.
    2.  **Database Schema:**
        * Define/update a PostgreSQL table to store patterns. Storing the complex `patternState` as a JSONB column is highly recommended for flexibility.
        * Table: `user_patterns`

            ```sql
            CREATE TABLE user_patterns (
                pattern_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
                user_id UUID NOT NULL, -- Foreign key to users table
                design_name VARCHAR(255) NOT NULL,
                garment_type VARCHAR(100), -- Extracted from pattern_data for easier querying
                pattern_data JSONB NOT NULL, -- Stores the entire patternState object
                version VARCHAR(50),         -- Version of the pattern_data structure
                created_at TIMESTAMPTZ DEFAULT NOW(),
                updated_at TIMESTAMPTZ DEFAULT NOW(),
                -- Potentially add thumbnail_url or other quick access metadata
                CONSTRAINT fk_user
                    FOREIGN KEY(user_id)
                    REFERENCES app_users(user_id) -- Assuming an app_users table
            );

            -- Index for faster lookup of patterns by user
            CREATE INDEX idx_user_patterns_user_id ON user_patterns(user_id);
            -- Index for searching by garment_type
            CREATE INDEX idx_user_patterns_garment_type ON user_patterns(garment_type);
            -- GIN index for querying inside the JSONB data if needed frequently
            -- CREATE INDEX idx_user_patterns_pattern_data_gin ON user_patterns USING GIN (pattern_data);

            -- Trigger to update updated_at timestamp
            CREATE OR REPLACE FUNCTION update_updated_at_column()
            RETURNS TRIGGER AS $$
            BEGIN
               NEW.updated_at = NOW();
               RETURN NEW;
            END;
            $$ language 'plpgsql';

            CREATE TRIGGER update_user_patterns_updated_at
            BEFORE UPDATE ON user_patterns
            FOR EACH ROW
            EXECUTE FUNCTION update_updated_at_column();
            ```
    3.  **Data Handling:**
        * The backend service will receive the `patternState` JSON.
        * It will extract `designName` and `garmentType` from the JSON to populate the dedicated columns for easier searching/indexing.
        * The entire `patternState` (including its own `version` field defined in PD_PH1_US002) is saved into the `pattern_data` JSONB column.
        * `patternState.metadata.patternId` should be populated with the `pattern_id` generated by the database upon first save. Subsequent saves would use this ID.
    4.  **Frontend:** The "Save Pattern" action will call this API endpoint. Update `patternState.metadata.patternId`, `createdAt`, `updatedAt` with the response from the server.
* **Acceptance Criteria (Testing & Validation):**
    1.  Given a user has defined a pattern (populating `patternState`).
    2.  When the user clicks "Save Pattern".
    3.  Then an API call is made to the backend with the `patternState`.
    4.  Verify a new record is created in the `user_patterns` table in PostgreSQL.
    5.  The `pattern_data` column in the new record must accurately store the `patternState` JSON.
    6.  `design_name`, `garment_type`, `user_id`, `version` fields are correctly populated.
    7.  The user receives a success confirmation. Subsequent saves update the existing record and its `updated_at` timestamp.
