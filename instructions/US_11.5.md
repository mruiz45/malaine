
**User Story 11.5**

1.  **Titre :** Implémenter la Génération de Diagrammes de Points (Stitch Charts) pour les Motifs de Tricot/Crochet
2.  **Objectif :** En tant que moteur de génération de patron, pour un motif de point sélectionné (issu de la bibliothèque de points, US 8.1) qui est mieux représenté par un diagramme, je dois générer une représentation graphique de ce diagramme (stitch chart) en utilisant des symboles standards.
3.  **Description :** Cette User Story se concentre sur la création de la logique pour convertir les instructions rang par rang d'un motif de point (ou une définition de diagramme préexistante) en une grille graphique de symboles. Les diagrammes de points sont cruciaux pour les motifs complexes comme la dentelle, les torsades, ou certains motifs texturés. Cela correspond à une partie importante de la section 5.3 du PDF ("Diagrammes et Schémas").
4.  **Exigences Fonctionnelles :**
    * FR1 : Le système doit pouvoir accéder aux détails d'un motif de point sélectionné via son ID (depuis la table `stitch_patterns`, US 8.1), qui devrait idéalement contenir une représentation structurée des instructions ou une définition de diagramme (par exemple, un tableau de chaînes de caractères où chaque chaîne est un rang et chaque caractère un symbole, ou une structure JSON plus formelle).
    * FR2 : Définir un ensemble de symboles graphiques standards pour les actions courantes de tricot (maille endroit, maille envers, jeté, 2m ens end, ssk, symboles de torsades, etc.) et de crochet (maille en l'air, maille serrée, bride, etc.). Ces symboles doivent être clairs et distincts.
    * FR3 : Implémenter une logique pour convertir la définition du motif de point en une grille de ces symboles. La grille représente les mailles et les rangs.
    * FR4 : Le diagramme généré doit inclure une légende expliquant chaque symbole utilisé dans ce diagramme spécifique.
    * FR5 : Les diagrammes doivent correctement indiquer le sens de lecture des rangs (par exemple, pour le tricot à plat, les rangs endroit de droite à gauche, les rangs envers de gauche à droite) et la numérotation des rangs et des mailles (typiquement pour un bloc de répétition).
    * FR6 : La sortie de cette fonctionnalité sera une structure de données représentant le diagramme (par exemple, un tableau 2D de clés de symboles, plus la légende) qui pourra ensuite être rendue graphiquement (dans US 11.6).
    * FR7 : Le système doit pouvoir gérer les "pas de maille" (no stitch) dans les diagrammes de dentelle où le nombre de mailles peut varier d'un rang à l'autre au sein d'une répétition.
5.  **Guide d'Implémentation Technique :**
    * **Modèles de Données :**
        * La table `stitch_patterns` (US 8.1) doit être enrichie (ou sa structure `chart_symbols` JSONB précisée) pour stocker la définition du diagramme de manière exploitable. Exemple de structure pour `chart_symbols`:
            ```json
            // Pour un motif de 4 mailles x 4 rangs
            {
              "width": 4, // en mailles
              "height": 4, // en rangs
              "grid": [ // Rang 0 (souvent Rang 1 dans les patrons)
                ["k", "p", "k", "p"], // k = knit, p = purl
                ["yo", "k2tog", "k", "k"], // yo = yarn over, k2tog = knit 2 together
                ["k", "p", "k", "p"],
                ["ssk", "yo", "p", "p"]  // ssk = slip slip knit
              ],
              "legend": [ // Liste des symboles utilisés dans CE diagramme
                {"symbol_key": "k", "definition": "Maille endroit (knit on RS, purl on WS if chart shows RS only)", "graphic_ref": "symbol_knit.svg"},
                {"symbol_key": "p", "definition": "Maille envers (purl on RS, knit on WS if chart shows RS only)", "graphic_ref": "symbol_purl.svg"},
                {"symbol_key": "yo", "definition": "Jeté (yarn over)", "graphic_ref": "symbol_yo.svg"},
                {"symbol_key": "k2tog", "definition": "2 mailles ensemble à l'endroit (knit 2 together)", "graphic_ref": "symbol_k2tog.svg"},
                {"symbol_key": "ssk", "definition": "Surjet simple (slip, slip, knit)", "graphic_ref": "symbol_ssk.svg"},
                {"symbol_key": "no_stitch", "definition": "Pas de maille (no stitch - for lace shaping)", "graphic_ref": "symbol_nostitch.svg"} // Souvent représenté par une case grisée
              ],
              "reading_direction_rs": "right_to_left", // Pour tricot à plat, rangs endroit
              "reading_direction_ws": "left_to_right"  // Pour tricot à plat, rangs envers
            }
            ```
        * Une table `standard_stitch_symbols` pourrait stocker la définition globale de tous les symboles supportés par le système, leurs graphiques SVG/image, et leur signification.
            ```sql
            CREATE TABLE IF NOT EXISTS standard_stitch_symbols (
                symbol_key VARCHAR(20) PRIMARY KEY,
                craft_type VARCHAR(10) NOT NULL CHECK (craft_type IN ('knitting', 'crochet')),
                description TEXT NOT NULL,
                graphic_asset_url TEXT NOT NULL, -- URL/path to an SVG or image file for the symbol
                notes TEXT
            );
            ```
    * **Logique/Traitement :**
        * Un service `StitchChartGeneratorService` prendrait l'ID d'un `stitch_pattern`.
        * Il récupère la définition du diagramme (`chart_symbols` JSONB) de la base de données.
        * Si `chart_symbols` n'est pas directement stocké mais que des instructions écrites le sont, une logique (plus complexe, potentiellement une US future) pourrait tenter de *convertir* des instructions écrites simples en une grille de diagramme. Pour cette US, nous supposons que `chart_symbols` est disponible ou que le motif est simple et que sa grille peut être construite à partir des `instructions_written`.
        * Le service valide la grille et s'assure que tous les symboles utilisés dans la grille ont une définition dans la légende (en les croisant avec `standard_stitch_symbols`).
        * La sortie est une structure de données propre (la grille, la légende spécifique au diagramme) prête à être rendue.
    * **Points d'Intégration :**
        * Utilise les données de la table `stitch_patterns` (US 8.1).
        * La sortie sera consommée par US 11.6 pour l'affichage web et l'inclusion PDF.
        * Fait partie du `CorePatternCalculationEngine` ou un service appelé par lui lorsqu'un patron utilise un diagramme.
6.  **Critères d'Acceptation (Validation Fonctionnelle) :**
    * AC1 : Pour un motif de point simple (ex: Côtes 2x2 sur 4 mailles et 1 rang) dont la définition de diagramme est stockée, le système génère une structure de données représentant la grille (ex: `[["k","k","p","p"]]`) et la légende correspondante.
    * AC2 : Pour un motif de dentelle simple utilisant des "jetés", "2m ens", et "pas de maille", la structure de données du diagramme inclut correctement ces symboles et leur légende.
    * AC3 : La légende générée contient uniquement les symboles utilisés dans le diagramme spécifique.
    * AC4 : Les informations sur le sens de lecture des rangs sont incluses si applicables (pour tricot à plat).
7.  **Stratégie de Test :**
    * **Tests Unitaires :**
        * Tester la fonction de récupération de la définition de diagramme pour un `stitch_pattern` donné.
        * Tester la fonction de construction de la légende spécifique au diagramme.
        * Tester la gestion des symboles "no_stitch".
        * Valider un diagramme simple (ex: damier de 2x2 mailles endroit/envers) avec sa grille et sa légende.
    * **Tests d'Intégration :**
        * S'assurer que le service peut correctement récupérer les définitions de symboles depuis `standard_stitch_symbols` pour construire la légende.
        * Vérifier que les `stitch_patterns` issus de la bibliothèque (US 8.1) fournissent des données compatibles avec ce générateur.
    * **Tests de Non-Régression :**
        * S'assurer que la sélection de points et la génération d'instructions textuelles pour les points simples (sans diagramme) ne sont pas affectées.
8.  **Hypothèses/Pré-conditions :**
    * La table `stitch_patterns` peut stocker une définition de diagramme structurée (ex: JSONB `chart_symbols`).
    * Une bibliothèque de symboles graphiques standards (fichiers SVG ou images de haute qualité) est disponible et référencée dans `standard_stitch_symbols`.
    * Pour cette US, nous nous concentrons sur la *génération de la structure de données* du diagramme. Le rendu graphique effectif sera traité dans la US suivante (11.6).
9.  **Composants Système Impactés (Illustratif) :**
    * Backend : Nouveau service `StitchChartGeneratorService.java`.
    * Base de Données : Évolution de la table `stitch_patterns` pour inclure `chart_symbols` de manière structurée ; nouvelle table `standard_stitch_symbols`.
